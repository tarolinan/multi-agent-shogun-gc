### 2. 検証環境の設計（2つの場の使い分け）

V3では「インフォマートをPhase 0に使う」としたが、就業時間中の成果物はFLUXに帰属するため不可。修正した設計:

  - ここで取れる実証データの種類と限界

- 2.2 インフォマート案件でのサイレント検証
  - 殿はPMとして参画。CPDDの名前は出さない

  - 表と裏の差分 = CPDDの検証データ（人間が見落としたものをCPDDが拾えたか）
  - 持ち出すのは「件数」「的中率」等の数字だけ。コード・チケット・画面は持ち出さない
  - **重要な制約**: 就業時間中の成果物はFLUX帰属。CPDDの検証作業は個人の時間で行う
- 2.3 「教えてくれ」と言われた時の論理武装
  - インフォマートで成果が出れば「やり方を教えろ」と必ず言われる

  - 公開レイヤーは手動だと面倒 → 契約終了後に「自動化の仕組みがある」と提案する伏線
- 2.4 2つの検証環境から得られるデータの統合方法

### 3. 「方法論」ではなく「プロダクト」としてのCPDD

V3までのCPDDは「人間が使う方法論」として設計されていた。これを修正する。

- 3.1 CPDDの正しいフレーミング
  - コードを食わせたら、AIが勝手に依存グラフを作る
  - 変更要求を投げたら、AIが勝手に影響範囲を出す
  - 人間は「ここ変えたい」と言うことと、AIの提示に対して「やる/やらない」を判断するだけ
  - これはClaude Codeが今やっていることと同じ構造（コンテキスト管理の自動化）
- 3.2 「方法論」と「プロダクト」の商品設計の違い
  - 方法論を売る: 人に教える必要がある→スケールしない→殿の時間が売上上限
  - プロダクトを売る: AIが動く→殿がいなくても回る→スケールする→億への道
  - ただしプロダクト化には技術検証が前提（セクション1が最優先の理由）
- 3.3 プロダクト化のロードマップ
  - Phase 1: プロトタイプ（殿が手動で回す。動くかどうかの検証）
  - Phase 2: ツール化（CLIまたはWebで半自動化。大里LMSで使う）
  - Phase 3: プロダクト化（殿がいなくても動く。外販可能）
  - 各フェーズの技術要件と期間見積もり
- 3.4 「コンテキスト管理 = brownfield変更管理」の構造的類似性
  - LLMのコンテキストウィンドウ管理と、レガシーコードの変更管理は同じ問題
  - 全部読み込む = モノリシック = 破綻する
  - 必要な部分だけロードする = モジュラー = スケールする
  - 依存グラフを辿って関連コンテキストだけを特定する = CPDD = LLMのコンテキスト管理
  - この類似性は、技術者への説明に使えるか？ Zenn記事や登壇のネタになるか？

## CPDD技術概要（シンプル版）

やりたいこと:
1. レガシーコードを食わせる → AIが自動で依存グラフを作る
2. 「ここを変えたい」と言う → AIが自動で影響範囲を出す
3. 人間は判断するだけ

依存関係の種類（5種類、段階的に実装）:
1. コード構造の依存（import、関数呼び出し、型参照）← まずここから
2. ビジネスルールの依存（設定値、条件分岐、業務ロジック）← 次にここ
3. 実行時の振る舞い（副作用、状態変更、外部API呼び出し）← 最も難しい
4. テストの対応関係（どのテストがどのコードをカバーしているか）
5. 承認・証跡（変更に必要な承認フロー、ドキュメント更新）

影響範囲の確度:
- 確実に影響する（自動対応可能）
- たぶん影響する（人間レビュー必要）
- 影響するかもしれない（ノイズの可能性あり）

競合状況:
- Fujitsuの「AI活用」は実態grep置換。依存グラフなし
- Kiroは既存アプリにも対応を広げているが、変更影響分析・承認証跡はない
- brownfieldでの本物のAI変更影響分析 = 世界で誰もやっていない
</subject_background>
