## 0. 一枚結論（200字以内）

V4の中心は、CPDDを売ることではなく、まず「実コードで本当に使えるか」を40件前後の評価ケースで判定することです。開発資産は大里LMSで作り、インフォマートは数字だけを取る比較観測に限定する。Q3末までにGo/No-Goを決め、動いたらプロダクト化、動かなければ即ピボットします。

## 1. CPDD技術検証戦略

### 1.1 何を検証すべきか

**結論**: 検証対象は2つ。①依存グラフの正しさ、②そのグラフを使った影響範囲追跡の当たり方。評価は「見つけた数」だけでは足りず、「見落としが少ないか」「ノイズが多すぎないか」「調査時間が減るか」まで見る必要がある。

V4で見るべき指標は5つ:
① file単位の precision / recall / F1
② function単位の precision / recall / F1
③ critical miss rate（最初の予測に入っておらず、後から追加された重要箇所の割合）
④ candidate inflation ratio（実際の影響件数に対して候補を何倍出したか）
⑤ 調査時間（1件あたり何分で候補一覧とチェックリストを作れたか）

真値の定義: ①実際に修正したfile/function、②変更で落ちた・更新したテスト、③レビューで後から追加した対象、④リリース後2週間以内の手戻り箇所

### 1.2 段階的検証の設計

**結論**: 実装順は 1→2→4→3→5 に変更。3番目は「実行時の振る舞い」ではなく「テストの対応関係」。

4段階:
- 第1段階: 主言語1つ・単一リポジトリで、コード構造だけの追跡。15ケース
- 第2段階: ビジネスルールを追加、同じケースでF1改善を測定。15〜20ケース
- 第3段階: テスト対応を追加、「どのテストを回すべきか」を出せるかを検証。10ケース
- 第4段階: 実行時の振る舞い。副作用・隠れた状態変更。5〜10の重要シナリオのみ

### 1.3 プロトタイプの技術的実現可能性

**結論**: 既存技術の組み合わせで十分作れる。難所は「実行時の振る舞い」だけ。

技術スタック:
- Tree-sitter: 増分解析で構文木作成、複数言語対応
- CodeQLの考え方: call graph / data flow / path query
- LLM: ビジネスルール候補をJSON制約で抽出
- OpenTelemetry: 実行時traceの後付け（LMSのみ）

各依存関係の解ける確度:
| 種類 | 静的解析のみ | 動的証跡あり |
|------|------------|------------|
| コード構造 | 80〜90% | - |
| ビジネスルール | 55〜70% | - |
| テスト対応 | 70〜85% | - |
| 実行時の振る舞い | 25〜40% | 60〜75% |
| 承認・証跡 | 80〜90% | - |

最小構成: ①Tree-sitterでノード抽出 → ②symbol table + edge作成 → ③LLMでビジネスルール候補抽出 → ④request-rule対応付け → ⑤深さ制限つき再帰追跡 → ⑥confidence bucket付きで返す

### 1.4 検証の成功基準

Go判定基準:
- ケース数: LMS 25件 + インフォマート 15件 = 40件最低ライン
- file単位: precision 75%以上、recall 85%以上、F1 0.80前後
- function単位: F1 0.65以上
- high-confidence precision: 90%以上
- 調査時間: 50%以上短縮（30〜45分 → 10〜15分）
- candidate inflation: 実影響の2倍以内
- critical miss rate: 5%未満

baseline: ①ripgrep文字列一致、②import検索、③caller/calleeの1-hop、④人間の簡易AI検索prompt
CPDD条件: baselineより F1を15ポイント以上、recallを20ポイント以上上回ること

### 1.5 検証が失敗した場合の撤退判断

3つのGate:
- **Gate 1** (40時間・15ケース): コード構造だけでfile recall 80%未満 → ピボット
- **Gate 2** (80時間・30ケース): ビジネスルール追加でF1が5ポイント以上改善しない or 3回で10ポイント以上ブレ → ビジネスルール自動化は時期尚早
- **Gate 3** (120時間・40件): file F1 0.72未満、high-confidence precision 85%未満、critical miss rate 10%超、時間短縮30%未満 → 「自動影響範囲」不成立

ピボット先:
1. コード構造 + テスト対応だけの「回すべきテスト選定ツール」
2. 「変更レビュー準備・承認証跡生成ツール」
3. プロダクトをやめて「brownfield変更統制のmanaged service」

## 2. 検証環境の設計

### 2.1 大里LMS案件での正式検証

- 6スプリント × 3〜5件 = 18〜30件 + 人工変更10件 = 28〜40件
- 保存可能: code、trace、prompt、graph、diff全部
- 限界: 企業大規模システムより隠れた依存が少ない
- 契約: ツール再利用権と匿名化指標公開権を明文化

### 2.2 インフォマート案件でのサイレント検証

プロトコル:
- 週1件、最大2件
- 表: エンジニアが手動で影響範囲洗い出し
- 裏: 殿が個人の時間でCPDD実行
- 保存: hash化したケースID + 数値のみ。コード・チケット・画面・prompt・graphは保存しない
- 制約: コード外部複製禁止、embedding/graph永続化禁止、model学習禁止

### 2.3 「教えてくれ」と言われた時の論理武装

公開レイヤー（5手順）:
1. 変更要求を1文で固定
2. 影響候補を「直接変更」「連鎖確認」「周辺更新」の3分類
3. 3分類ごとにAIへ候補探索
4. 人間が10〜15分レビュー
5. ticketにチェックリストとして残す

教えないもの: ①依存グラフの自動構築 ②再帰的影響追跡 ③確度の自動判定

言い方: 「共有できるのは運用手順です。自動判定の仕組み自体は、現契約の対象外です。」

### 2.4 2つの検証環境のデータ統合

LMS = 技術妥当性、インフォマート = 現場有用性の2軸で別々に判定。
Go条件: 両方通った時だけ。足し算ではなく、両方必要。

## 3. 「方法論」ではなく「プロダクト」としてのCPDD

### 3.1 正しいフレーミング

MVP出力4点: ①影響候補一覧 ②なぜ候補に入ったかの経路 ③確度 ④未解決の確認ポイント

### 3.2 方法論 vs プロダクトの違い

- 方法論: 1チーム導入に4〜8時間。10社で時間が詰まる。殿の時間=売上上限
- プロダクト: 導入30〜90分。変更1件のコスト=API+計算資源。スケール可能

### 3.3 プロダクト化ロードマップ

- Phase 1 (8〜10週/80〜120時間): 主言語1つ、CLI、コード構造+ビジネスルール
- Phase 2 (10〜12週/100〜140時間): test mapping、簡易UI、diff入力、checklist出力
- Phase 3 (16〜24週/カレンダー6〜9ヶ月): 権限管理、監査ログ、閉域配備、外販可能

配備形態: multi-tenant SaaSではなく顧客ごとの閉域配備

### 3.4 コンテキスト管理 = brownfield変更管理の類似性

- 技術者向け発信には使える
- 経営向けの主メッセージには向かない（経営が買うのは「変更漏れが減る」「手戻りが減る」）
- 発信タイミング: Gate 2通過後

## 6. レッドチーム
