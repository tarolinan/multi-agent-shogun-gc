# GPT-5.4 Pro v2 結果2（Claude Code CLIプロンプト）

## 1. 結論の一文

主戦略は **B'** です。つまり、**外向きブランドは「CPDD」を捨て、思想・依存メタスキーマ・最小テンプレートだけを公開し、brownfield の解析器・Change Propagation engine・評価データ・導入 playbook は閉じたまま、FLUX在籍中に有償診断を2件、300万円級 Pilot を1件取り、その Pilot の着手金と事例化権を持って辞める**、です。
理由は、方法論/CLIの一般層はすでに GitHub Spec Kit の MIT 公開、Kiro の spec-driven 商用化、富士通の全工程AI開発基盤の発表で crowded であり、OpenHands も OSS は個人ローカル、商用はチーム/クラウド/ガバナンスに明確に線を引いているからです。さらに「CPDD」は Jaana Dogan が別概念として既に使っており、外向き名称としては不利です。あなたの勝ち筋は、**日本企業の brownfield × 変更統制 × traceability** に絞った有償変革です。

## 2. 前回からの変更点サマリ

1. **「影の時給25,000円」は理屈としては崩れた。** 年俸制で副業の金銭的機会費用がゼロだからです。ただし、後で計算すると、**注意力基準でも実務上の成立ラインは結局 2.5〜3.0万円/深い注意1時間**あたりに戻ります。数字は近いが、根拠が変わりました。

2. **入口商品の価格下限は少しだけ下がる。** ただし「小さい案件なら何でもよい」ではありません。下がるのは、**完全定型・会議少・アフターサポートなし**の入口だけです。カスタム仕事の下限はほぼ下がりません。

3. **「個人信用ゼロ」は半分だけ正しい。** 集客信用はゼロではありません。Zenn・X・登壇・コミュニティで、面談を作る力はもうあります。ゼロに近いのは**調達信用・実行信用**です。だから最初の2件は「売上」より「証拠」の意味が大きいです。

4. **辞める判断はカレンダーではなく proof に寄せる。** Aのような「2026年10月だから辞める」は弱い。Bのような「Pilotが取れたら辞める」が正しい方向です。ただし本当に必要なのは **Pilot受注 + 着手金 + 事例化権 + 次案件の見込み** です。

5. **公開方針は"何を開けるか"より"いつ開けるか"が変わる。** Aider のように stars / installs を前面に出す reach は作れても、Kiro や OpenHands のような明確な pricing / enterprise package がなければ cash capture になりません。だから **OSS は offer と導線を作ってから** です。

## 3. 成立ラインの再定義

前提として、厚労省の副業・兼業ガイドラインは、副業促進を認めつつも、**長時間労働と健康確保**を重く見ています。あなたのケースでは、金よりもここが本当の制約です。つまり判定軸は「本業の給与を代替するか」ではなく、**週明けの集中力を壊さずに、どれだけ手残りを積めるか**です。

### 新しい式

**注意力1時間あたり手残り**

**(案件粗利 × 回収確率 + 次案件期待粗利 × 波及確率 + 事例化価値 + 再利用資産価値 − 将来保守債務 − 月曜毀損期待値)** ÷ **深い注意時間**

ここでいう「深い注意時間」は、設計・営業・提案・実装で脳を強く使う時間です。雑務時間ではありません。

### 実務ライン

| ゾーン | 基準 |
|---|---|
| 成立 | **3万円/h以上**、または **2万円/h以上 + 事例/紹介/再利用資産が残る** |
| 条件付き成立 | **2〜3万円/h** だが、warm lead で、会議少、保守なし |
| 不成立 | **2万円/h未満**、または weekday SLA、または保守が膨らむ |

### 前回の価格階段をこの基準で再計算すると

以下は **個人売上の最終手残り率を 55%** と仮置きした試算です。税や法人化で上下するので、これは仮説です。

| 商品 | 価格 | 深い注意時間 | 手残り/注意h | 判定 |
|---|---|---|---|---|
| 20万レビュー | 20万 | 8h | 約1.4万/h | 単体では弱い |
| 20万レビュー + 20%で100万診断に繋がる | 期待40万 | 8h | 約2.8万/h | warm leadなら成立 |
| 100万診断 | 100万 | 18h | 約3.1万/h | **最良** |
| 300万Pilot | 300万 | 60h | 約2.8万/h | 事例化権があるなら成立 |
| 50万/月伴走 | 50万 | 10h | 約2.8万/h | SLA厳守なら成立 |

**結論**:
前回の「25,000円/h」は給与代替のロジックとしては誤りでした。ですが、**注意力と翌週パフォーマンス毀損を織り込むと、実務上の境目は結局 25,000〜30,000円/h 近辺に再出現**します。
だから、小さい案件は合理化されるが、**安いカスタム案件は依然として不合理**です。

* **Bが最適。** ただし私は B を **B'** に補正します。
  **Pilot 1件受注**だけではなく、
  1. **50%着手金**、
  2. **事例化権**、
  3. **次の100万診断2件見込み**

* **Cは数字が良く見えるが、実は"安全すぎて発火しない"危険**があります。今の scarce attention で年1,000万副業は簡単ではありません。
* **Dは3年キャッシュの期待値が高い。** ここが最大の迷いどころです。ただし、CPDDを**本業化する確率**は最も低いです。

**断言すると、CPDD事業を作る目的なら B'、3年キャッシュ最大化だけが目的なら C/D です。**

## 5. FLUX看板 vs 個人の価格・信用分析

### まず分解

あなたの「個人信用」はゼロではありません。
ただし3層に分かれます。

* **集客信用**: ある
  Zenn、X、登壇、コミュニティ、LinkedInスカウトで面談を作る力はある。
* **課題理解信用**: かなりある
  SI全工程、PM、マルチベンダー、AI実装、登壇実績で十分ある。
* **調達信用 / 実行信用**: 低い
  ここだけは CPDD の有償事例がない。

つまり「個人渡辺はゼロ」は強すぎます。
正しくは、**meeting を作る信用はあるが、300万超の PO を出させる信用はまだない**です。

  理由は simple で、金銭機会費用がゼロだからです。今は cash pressure なしで proof を買いにいける。
* ただし **価格を下げるより、scope を狭める** のが正解です。

1. **匿名でもよいので brownfield change-control の実演動画を作る**
2. **CPDDの売り物を"CLI"ではなく"変更影響一覧と対応計画"に変換する**
3. **最初の2件は必ず testimonial / case-rights 条項を入れる**
4. **提案資料を IPA 語彙で作る**

厚労省のモデル就業規則では blanket な副業禁止規定は削除されている一方、実際には会社ごとの就業規則、労働時間管理、健康確保、合意書が重要です。ここは一般論ではなく、**契約・規程確認が前提**です。

## 6. 商品階段の再設計

Kiro は自然文から requirements / design / tasks を作り、Spec Kit も agent-generic にその層を公開しています。つまり、売るべきは「spec を作ること」ではありません。売るべきは、**変更が入った瞬間に、どの artifact と test が影響を受けるかを出せること**です。さらに IPA 共通フレームは、ライフサイクル全体の役割・作業項目・共通言語を重視し、調達ガイドでも契約変更管理を重要視しています。あなたはそこに寄せた方が強いです。

### 修正版の階段

| 段階 | 商品 | 価格 | 役割 |
|---|---|---|---|
| 0 | 記事・登壇・動画デモ | 無料 | 集客 |
| 1 | 変更影響レビュー | 20〜30万 | 入口、課題顕在化 |
| 2 | CPDD適用診断スプリント | 80〜120万 | **主力**、提案・証拠作り |
| 3 | Intro Pilot | 250〜350万 | 個人初期の実証 |
| 4 | Full Pilot | 400〜500万 | 2事例後の標準 |
| 5 | 導入伴走 | 40〜80万/月 | 継続収益 |
| 6 | 研修/内製化支援 | 30〜80万/回 | 補助商品 |

### 何が変わったか

* **20万レビュー → 残す**
  ただし **warm lead 限定**。cold に広く売る商品ではありません。
* **100万診断 → むしろ主力化**
  ここが最も attention ROI が高いです。
* **400万Pilot → 二段化**
  個人初期は 250〜350万の Intro Pilot。2事例後に 400〜500万へ。
* **入口商品の最低ラインは"価格"より"boundedness"で決める**
  10〜15万でも、完全定型・非同期・4h以内なら合理的。
  逆に 20万でも、会議だらけなら不合理。

**結論**:
前回の **20万レビュー → 100万診断 → 400万Pilot** の背骨は維持でよいです。
修正点は、
* 20万レビューを **funnel-only** にすること
* Pilot を **Intro / Full** に分けること
* 主役を **100万診断** に据えること
です。

## 7. 資産別公開マップ（変更部分のみ）

前回の **open thesis / closed engine / paid transformation** 自体は正しいです。
今回変えるのは **公開順序** です。
在籍中の scarce attention を考えると、**offer page と CTA ができる前に OSS を出すのは早い**です。

| 資産 | 人を集める | 信頼を作る | お金を取る | 模倣を防ぐ | 判断 |
|---|---|---|---|---|---|
| 思想記事・登壇・比較論 | ◎ | ○ | × | × | 今すぐ公開 |
| toy brownfield demo repo + 動画 | ◎ | ◎ | △ | × | 今すぐ公開 |
| 依存メタスキーマ | ○ | ◎ | △ | × | 1件目の診断後に公開 |
| starter templates / init CLI | ○ | ○ | △ | × | offer page完成後に OSS |
| brownfield extractor / adapters | × | ○ | ◎ | ◎ | 非公開 |
| propagation planner / evaluator | × | ◎ | ◎ | ◎ | 非公開 |
| 実案件 corpus / failure taxonomy | × | ◎ | ◎ | ◎ | 非公開 |
| 提案書、ROI計算、導入 playbook | × | ◎ | ◎ | ○ | 有料サービス内 |
| SSO / VPC / on-prem packaging | × | ○ | ◎ | ○ | **後回し** |
| SaaS seats / multi-tenant cloud | △ | △ | ◎? | × | **やらない** |

OpenHands が実際にやっているのも、**個人が自分のPCで使う層を open にし、チーム運用・クラウド・VPC・SSO・support を commercial に切る**やり方です。あなたもそれに近い線がよいです。

そして、ここで大事なのが **Aider の教訓** です。公開面では Aider は stars / installs / benchmarks を強く見せていますが、Kiro や OpenHands は pricing / enterprise package を正面に出しています。つまり **reach と capture は別物** です。公開物はすべて、**診断への CTA** に接続して初めて意味があります。

## 11. 最大の反論

**最大の反論はこれです。**
**「B'で辞める必要が本当にあるのか。DかCの方が、3年キャッシュは強く、リスクも低い。しかも generic layer は既に混んでいる。CPDDを会社にしようとするより、給与を維持したまま side consulting にした方が合理的ではないか。」**

これは強い反論です。実際、**3年累計手取りだけ見れば C/D は B' を上回りうる** からです。

### この反論が成立する条件

1. 土日 delivery が **月曜パフォーマンスをほぼ落とさない**
2. 在籍中でも **年1,000万副業が現実的**
3. 企業案件が **週末中心でも回せる**
4. あなた自身の目的が **独立事業化ではなく cash 最大化** である

### その場合の代替戦略

その場合は **D/C ハイブリッド** が正解です。
つまり、
* 辞めない
* 公開は薄くやる
* CPDDは**商品ではなく delivery OS** として使う
* 高単価の side diagnostic / pilot だけ取る
* 独立は 2027年以降に再判定
です。

ただし私は、それでも **主戦略は B'** を推します。
理由は、あなたの目的が「副収入を増やすこと」ではなく、**自分の名前で勝てる CPDD 事業を作ること** だからです。D/C は cash には強いが、**事業化速度** に弱いです。
