## CPDDとは

CPDDは、ソフトウェアの変更要求に対して、影響範囲を自動で特定するプロダクトです。

やることはシンプルです:
1. コードベースを解析して依存グラフ（メタデータ）を自動生成する
2. 「ここを変えたい」と言われたら、依存グラフを辿って影響範囲を出す
3. 人間は結果を見て判断するだけ

## 解決すべき現実の問題

富士通で20年間、数千万STEP級の大規模電子カルテパッケージ（EGMAIN-GX, LifeMark-HX）の保守PMを務めた人物からのヒアリングに基づく、現場の現実です。

### 現状の影響範囲調査プロセス（人力）

1. 変更要求を受ける（例:「診療報酬改定で〇〇の点数が変わった」「画面Xで特定条件のときにエラーが出る」）
2. 起点のモジュールを特定する（これは比較的すぐできる）
3. grepで直接参照している箇所を探す
4. 見つかった箇所について、さらに以下を確認する:
   - そのモジュールが使っているマスターテーブルや設定を特定する
   - 同じマスターテーブル/設定を使っている別のモジュールを探す
   - これが「grepでは見えない間接依存」であり、最も時間がかかる部分
5. さらにその先を再帰的に追う
6. 影響範囲の一覧表を作成する

**所要時間**: ベテランPMでも大きな変更で1週間。新人にはそもそもできない。
**品質**: 誰がやってもMECE（漏れなし）にはならない。属人的で、人によってやり方が違う。

### なぜ難しいのか（具体的な難所）

**難所1: ソースコード外を媒介した間接依存**
モジュールAとモジュールBはコード上で直接呼び合っていないが、同じマスターテーブルを参照している。Aがマスターの意味を変えたらBが壊れる。call graphには出てこない。grepでは見えない。

**難所2: 設計書の陳腐化**
エンハンスを重ねるうちに設計書が実態と乖離する。設計書を鵜呑みにすると影響箇所を見落とす。結局grepとコード読みに頼る。

**難所3: 人によって書き方が違う、言語が混在**
同じシステム内でも、開発者ごとに命名規則やパターンが異なる。Java + SQL + シェルスクリプト + 設定ファイル + 画面定義等が混在。

**難所4: OOPの継承・オーバーライド**
基底クラスのメソッドをオーバーライドすると、実際の振る舞いが1箇所に書かれていない。人間の目視では合成が必要で、漏れやすい。

**難所5: 動かさないと分からない問題**
タイミングイシュー、非同期処理の副作用、実行順序依存。静的解析の原理的限界。

**難所6: 「まさかここに影響するとは」**
自分が触った場所と全く別の場所で障害が出る。因果関係が遠すぎて予測不能。最も多い事故パターン。

**難所7: 変更要求から調査起点への変換**
「月次請求で複数契約の顧客の計算がおかしい」→ じゃあどのテーブル、どのモジュール、どのコードパスを調べればいいのか？この変換自体がベテランの暗黙知。

## 重大な設計要件: greenfield/brownfield統一アーキテクチャ

CPDDは2つの入口を持ちます:

**入口1: greenfield（0→1で新規開発する時）**
- コードを書きながら、将来の影響範囲調査に必要なメタデータも同時に生成する
- 設計書とは別に、メンテナンス用のメタデータを作る
- コードが変わるたびにメタデータも自動更新される

**入口2: brownfield（既存レガシーコードに適用する時）**
- 既存コードをリバースエンジニアリングして、同じメタデータを自動生成する
- 設計書は信用しない。コードが唯一の真実。
- 数千万STEPのコードに対しても現実的な時間で生成可能であること

**重要**: 入口1と入口2の出力フォーマットは同一であること。どちらの入口から入っても、同じ依存グラフ形式に収束し、同じCPDDエンジンで影響範囲追跡ができること。
</system_context>

## 必須出力構造

### 0. 一枚結論（200字以内）
V5の最も重要な設計判断を1パラグラフで。

### 1. メタデータ仕様の設計

CPDDが生成・保持すべきメタデータの完全な仕様を設計せよ。

- 1.1 メタデータの全体構造（どのようなデータをどのような形式で持つか）
- 1.2 各メタデータ項目の詳細定義
  - 何を格納するか
  - なぜそれが影響範囲調査に必要か
  - どのような粒度で持つか（ファイル単位？メソッド単位？行単位？）
  - データの具体的なスキーマ（テーブル定義 or JSONスキーマ）
- 1.3 難所1（マスター/設定を媒介した間接依存）を解くために必要なメタデータ
- 1.4 難所4（継承・オーバーライド）を解くために必要なメタデータ
- 1.5 難所7（変更要求→調査起点の変換）を解くために必要なメタデータ
- 1.6 メタデータのストレージ設計（RDBMS? Graph DB? ファイル? 規模による選択基準）

### 2. greenfield時のメタデータ自動生成

0→1で新規開発する時に、コードを書きながらメタデータを自動生成する仕組みの設計。

- 2.1 どのタイミングで生成するか（コミット時？ビルド時？リアルタイム？）
- 2.2 どのツール/技術で生成するか（AST解析、静的解析、LLM、それらの組み合わせ）
- 2.3 開発者に追加負担をかけない設計（自動でなければ陳腐化する）
- 2.4 コード変更時のメタデータ増分更新の仕組み
- 2.5 具体的な実装アーキテクチャ（CLI? CI/CDパイプライン? IDE plugin?）

### 3. brownfield時のリバースエンジニアリング

既存レガシーコードからメタデータを自動生成する仕組みの設計。

- 3.1 数千万STEPのコードに対して現実的な時間で解析する方法
- 3.2 言語混在（Java + SQL + シェル + 設定ファイル + 画面定義）への対応
- 3.3 call graph自動生成の技術的手法と精度
- 3.4 data access map（テーブル×モジュールのマトリクス）自動生成の技術的手法
  - SQL文の解析（直書きSQL、ORM経由、動的SQL、ストアドプロシージャ）
  - 設定ファイルの参照解析
  - 環境変数、外部連携の検出
- 3.5 型階層（継承・オーバーライド）の自動解析
- 3.6 設計書が陳腐化している前提での解析戦略（コードのみから抽出）
- 3.7 解析精度の限界と、それを補完する方法

### 4. 影響範囲追跡エンジンの設計

メタデータを使って、変更要求から影響範囲を自動特定するエンジンの設計。

- 4.1 追跡アルゴリズムの詳細設計
  - call graphの再帰追跡
  - data access map経由の間接依存追跡
  - 型階層を考慮したオーバーライド解決
  - 上記3つを組み合わせた統合追跡
- 4.2 追跡の深さ制限と確度判定
  - 再帰の深さによる確度の減衰をどう扱うか
  - 確実に影響する / たぶん影響する / 影響するかもしれない の分類ロジック
- 4.3 変更要求（自然言語）→ 調査起点への変換
  - LLMを使った変更要求の解釈
  - ビジネス概念からコード上のモジュールへのマッピング
  - このマッピングの精度をどう担保するか
- 4.4 出力フォーマット
  - 影響候補一覧の構造
  - なぜ候補に入ったかの経路説明
  - 確度の表示
  - 人間が判断しやすい表示形式
- 4.5 数千万STEPのコードベースでの性能要件
  - 追跡にかかる時間の目安
  - メモリ/ストレージの要件

### 5. 難所への個別解法

各難所に対する具体的な技術的解法を設計せよ。

- 5.1 マスター/設定を媒介した間接依存（難所1）
- 5.2 設計書の陳腐化（難所2）→ コードのみからの自動生成で解決する設計
- 5.3 人によって書き方が違う/言語混在（難所3）
- 5.4 継承・オーバーライド（難所4）
- 5.5 動かさないと分からない問題（難所5）→ 静的解析の限界と動的トレースの組み合わせ

## 技術的な前提知識

### 既存ツールでできること（CPDDが利用できるもの）

- **Tree-sitter**: 増分解析で構文木を生成。複数言語対応。高速
- **Language Server Protocol (LSP)**: 定義ジャンプ、参照検索、型情報の取得
- **CodeQL的アプローチ**: call graph、data flow、path queryの考え方
- **LLM（GPT-5.4 / Claude Opus）**: コードの意味理解、ビジネスルール推定、自然言語変換
- **OpenTelemetry**: 実行時のtrace記録（動的解析用）
- **Coverage.py等**: テストカバレッジ、どのテストがどの行を通るか

### CPDDが既存ツールと違うところ

- grep: 文字列一致のみ。意味的な依存を追えない
- LSP: 1ファイル内の参照は追えるが、マスターテーブル経由の間接依存は追えない
- IDE: 定義ジャンプはできるが、影響範囲の網羅的な列挙はしない
- Fujitsuの事例: grepレベルのテキスト置換をAIと詐称。依存グラフなし
- **CPDDの独自価値**: call graph + data access map + 型階層を統合した再帰的影響範囲追跡

### V4で決まった検証基準（参考）

- Gate 1 (40時間・15ケース): コード構造だけでfile recall 80%未満 → ピボット
- Gate 2 (80時間・30ケース): ビジネスルール追加でF1が5ポイント以上改善しない → 時期尚早
- Gate 3 (120時間・40件): file F1 0.72未満 → 「自動影響範囲」不成立
- 成功基準: file F1 0.80前後、high-confidence precision 0.90以上、調査時間50%以上短縮
