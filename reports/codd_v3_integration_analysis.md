**目的**: 9回分のGPT分析 + Opus独自分析を統合し、年収成長パスを含む実行可能な最終戦略を出す

## 0. 一枚結論

**9回分の分析が出揃い、3層が完成した。**

| 層 | 担当 | 内容 |
|----|------|------|
| ビジネス戦略 | V1-V3 + Opus | 何を・誰に・いくらで売るか |
| 技術検証戦略 | V4 | 売る前に本当に動くか証明する |
| 技術設計 | V5 | どう作るか（実装仕様レベル） |

**V4が全体を変えた最大の修正**: 「Phase 0でデータを取る→売る」ではなく「Phase 0でデータを取る→**Go/No-Go判定する**→通ったら売る」。検証なしに売る戦略は危険だった。

**V5が確定させたこと**: CPDDの正体は「1つの巨大なグラフDB」ではなく「同一フォーマットのメタデータ束（cpdd-bundle/v1）」。プロトタイプは4-5週間・28-40時間で作れる。

**両頭立て戦略（3/20確定）**: CPDDの検証は難易度の異なる2環境で段階的に行う。

| # | 環境 | 性質 | CPDD難易度 | 証明すること |
|---|------|------|-----------|-------------|
| 1 | 大里LMS | **Greenfield（0→1）** | 低い — 最初からCPDD前提で設計 | CPDDで0→1のプロダクト開発を効率化できるか |
| 2 | インフォマート | **Brownfield（既存エンハンス）** | 高い — モンスター級の既存コードをコンテキスト分解し波に乗せる | レガシーコードの変更影響を素早く把握しエンハンスを加速できるか |

**順序が重要**: まず易しいGreenfield（大里）で方法論を固め、次に難しいBrownfield（インフォマート）でエンタープライズ級の実証を行う。

売り文句は確定: 「承認可能な変更を、品質を落とさず、速く通す」。

## 1. V3がOpusを強化した点（採用確定）

### 1.1 Phase 0の実行計画 ★最重要

Opusは「インフォマートでCPDDを内部適用しろ」と言っただけ。V3はそれを**4段階の具体計画**にした。

| 期間 | やること | 成果物 |
|------|---------|--------|
| 4月前半 | Ticket Shadow Sheet作成 | 計測テンプレート（1件5分） |
| 4-5月 | ベースライン取得（10-12件） | 現行の影響調査時間・見落とし率 |
| 6-7月 | 1サービス群でCPDD-lite（10-15件） | Green/Amber/Gray判定の精度データ |
| 8-9月 | 3サービス以上に拡大（累計20-30件） | 匿名ケースメモ5p + 比較グラフ3枚 + 擬似リポジトリ |

**Opus評価**: 完璧。このまま採用。唯一の補足は「4月前半の10営業日」は甘い可能性がある — インフォマート着任直後は環境構築とオンボーディングで消える。Shadow Sheetは**4月3週目**開始が現実的。

### 1.2 Clean-room 3層モデル

| 層 | 内容 | 外部持ち出し |
|----|------|-------------|
| Client Vault | コード、設定、チケット、画面、実プロンプト | ❌ 絶対不可 |
| Derived Ledger | 件数、時間、精度の集計値 | ⚠️ 4条件クリアで可 |
| Synthetic Lab | 擬似リポジトリ、一般化パターン、合成データ | ✅ 営業・学術・記事用 |

**Opus評価**: 3層分離は正しい。4条件（固有名詞ゼロ・原文ゼロ・3件以上集計・元案件復元不能）も妥当。採用。

### 1.3 メトリクス体系

V3のKPI設計はOpusより遥かに具体的。買い手の稟議に載る言葉で組まれている。

**5群のKPI**:
1. **速度**: 影響調査初稿時間（-30〜50%）、承認資料準備時間（-50%）
2. **品質**: 見落とし手戻り件数（-20%）、差し戻し率、障害/ロールバック件数
3. **精度**: Green precision ≥0.85、Amber yield ≥0.40、Gray hit ≥0.15
4. **統制**: 要件-設計-実装-テスト-承認リンク完備率（≥80%）、証跡再取得時間（-50%）
5. **成立条件**: n≥20件、3サービス以上、2変更類型以上

**Opus評価**: 「LoC、トークン数、ストーリーポイントをKPIにするな」は正しい。買い手は「調査が速くなったか」「見落としが減ったか」「承認が通りやすくなったか」しか見ない。採用。

### 1.4 ADHD最適化オペレーション

V3の**トリガー型ワークフロー**はOpusの「固定スコープ×非同期×結果物納品」を運用レベルに落としたもの。

**注意力破綻の自動停止ルール**:
- 2指標超過 → 72時間営業停止
- 3指標超過 → 14日新規受注停止
- 4指標超過 → 30日縮退運転

**Opus評価**: 将軍システムに組み込めば自動検知可能。「感情ではなく運用指標で判断」は殿の気質に合う。採用。

### 1.5 提案書の言葉

V3がIPA共通フレーム語彙で組んだ5文:

1. 「変更要求を影響の調査分析へ落とし、処置案を承認可能な形にする方法」
2. 「AIで全部自動化することではなく、人が安全に変更を通せるようにすること」
3. 「コード生成の前に、要件・設計・実装・テスト・文書・承認資料のどこに波及するかを見る」
4. 「開発速度の話ではなく、変更管理のリードタイムと説明可能性の話」
5. 「SDDが新規開発を整えるなら、CPDDは既存変更の合意と承認を整える」

**Opus評価**: 全採用。特に文4と文5は殺し文句。CIO/IT子会社部長に刺さる。

## 7. V4の統合 — 技術検証戦略（ビジネス戦略を根本的に修正）

### 7.1 V4の核心的修正

V3までの暗黙の前提: 「CPDDは動く。問題はいつ・どう売るか。」
V4の修正: **「CPDDが本当に動くかまだ分からない。まず証明しろ。」**

これは正しい。売り物が動かなければ戦略もクソもない。

### 7.2 Go/No-Go判定（9月末）

**Go条件（全て満たす必要あり）**:

| 指標 | 基準 | 意味 |
|------|------|------|
| file precision | ≥75% | 出した候補が外れすぎない |
| file recall | ≥85% | 見落としが少ない |
| file F1 | ≥0.80 | 総合精度 |
| function F1 | ≥0.65 | メソッド単位でもそこそこ当たる |
| high-confidence precision | ≥90% | 「確実に影響する」と言ったものがほぼ正しい |
| 調査時間短縮 | ≥50% | 30-45分 → 10-15分 |
| candidate inflation | ≤2倍 | ノイズが多すぎない |
| critical miss rate | <5% | 致命的な見落としがほぼない |
| ケース数 | ≥40件（LMS 25 + インフォマート 15） | 統計的に有意 |

**Opus評価**: 基準は厳しめだが妥当。F1 0.80は高いが「安全に変更を通す」ためには必要。**critical miss rate <5%が最重要** — 見落としがあったら信頼が崩壊する。

**両頭立て戦略における25+15の意味**（3/20追記）:
- **LMS 25件 = Greenfield検証**。0→1でCPDD前提で作ったコードベースでの精度。ここが低ければ方法論自体に問題がある
- **インフォマート15件 = Brownfield検証**。既存モンスターコードをコンテキスト分解→波に乗せた上での精度。**商品として売る時の本番環境はこちら**
- Greenfield精度 > Brownfield精度は当然予想される。**売り物の信頼性はBrownfield側の数字で決まる**

### 7.3 3段階ゲート

| Gate | 投入時間 | ケース数 | 判定 |
|------|---------|---------|------|
| Gate 1 | 40h | 15件 | コード構造だけでfile recall 80%未満 → ピボット |
| Gate 2 | 80h | 30件 | ビジネスルール追加でF1改善 <5pt → ビジネスルール自動化は時期尚早 |
| Gate 3 | 120h | 40件 | 全指標チェック → Go/No-Go最終判定 |

**Opus評価**: ゲート方式は正しい。120時間投入して「ダメでした」が分かるのは、外部案件を取ってから「ダメでした」より100倍マシ。

### 7.4 失敗時のピボット先（V4の最大の価値）

V3まではピボット先が「managed service」しかなかった。V4は3つ出した:

1. **テスト選定ツール**: コード構造 + テスト対応だけで「どのテストを回すべきか」を出す。影響分析より狭いが、需要は確実にある
2. **変更レビュー準備・承認証跡生成ツール**: 変更対象の文書化と証跡パック生成に特化。技術的に容易
3. **Brownfield変更統制のmanaged service**: ツールは出さず、コンサルとして人力+AIで影響分析を提供

**Opus評価**: ピボット1（テスト選定）が最も商品化しやすい。CI/CDに組み込める、価格帯が低い（月額制可能）、需要が明確。**ピボットがあるから安心してGo/No-Goに臨める。**

### 7.5 成功確率（V4の主観確率）

| シナリオ | 確率 | 意味 |
|---------|------|------|
| 完全成功（5層全体が商品レベル） | 35% | CPDDフル版が売れる |
| 部分成功（2-3層が使えるレベル） | 45% | スコープ縮小して売る |
| 失敗（コード構造すら精度不足） | 20% | ピボット |

**Opus評価**: 完全成功35%は現実的。部分成功45%を合わせると**80%の確率で何かは売れる**。これは投資として十分。

### 7.6 V4がビジネス戦略に与える影響

**年収ロードマップの修正**:

V3版: 2026Q4に外販開始（確定的）
V4版: 2026Q4に外販開始（**Go判定通過が前提**）

確率加重すると:
- 2027年メインシナリオ（Go通過時）: 828万 → **変更なし**
- 2027年ピボットシナリオ（部分成功時）: テスト選定ツール月額5万×20社 = 年1,200万の可能性もある。ただし立ち上げに時間がかかるので2027年は300-500万レンジ
- 2027年失敗シナリオ: 受託+コンサルで600万

**確率加重2027年収**: 828×0.35 + 700×0.45 + 600×0.20 = **725万円**

→ V3の828万より保守的になるが、**ピボット先があることでリスクが管理可能になった**。

### 7.7 「方法論」vs「プロダクト」のフレーミング

V4が提起した重要な問い: **CPDDは方法論か、プロダクトか？**

| 軸 | 方法論 | プロダクト |
|----|--------|----------|
| スケール | 殿の時間 = 売上上限 | API+計算資源 = スケール可能 |
| 導入コスト | 4-8時間/社 | 30-90分/社 |
| 年収上限 | 2,000-3,000万 | 青天井 |
| 開発コスト | 低い | 高い（プロトタイプ28-40h + Phase 2-3） |

**Opus判断**: **最初は方法論として売り、検証データが揃ったらプロダクト化する。** 順番が逆だと失敗する。V5のプロトタイプは「プロダクト化の技術的実現可能性検証」として位置づける。

## 8. V5の統合 — 技術設計（実装仕様レベル）

### 8.1 V5の核心: 「メタデータ束」アーキテクチャ

旧設計（V1-V3）: 「統合依存グラフ（UDG）」= 1つの巨大なグラフDB
V5の修正: 「cpdd-bundle/v1」= **同一フォーマットのメタデータ束**

5種類のコアデータ:

| データ | 役割 | 旧設計との違い |
|--------|------|---------------|
| node | 影響範囲の候補（file/type/callable/db_table/config_key等） | ほぼ同じ |
| edge | 汎用の有向関係（calls/extends/implements等） | ほぼ同じ |
| resource_usage | **テーブル/設定を媒介した間接依存**（列・条件レベル） | ★新規。edgeに押し込まず専用テーブル |
| term / term_link | **自然言語→コード起点の変換索引** | ★新規 |
| runtime_observation | **実行時証跡** | ★新規。静的解析と分離 |

**Opus評価**: resource_usageの分離は正しい判断。テーブル名だけの一致では「同じマスターを見ているだけ」のノイズが爆発する。**列・条件レベルまで持つ**ことで、precision が劇的に上がる。

### 8.2 追跡エンジン — 優先度付き多点探索

V5の追跡アルゴリズムは3フェーズ:

```
Phase A: 要求文 → 起点候補（term indexで5個に絞る）
Phase B: 表面起点 → 実装起点（depth 3以内）
Phase C: 実装起点 → 影響候補（priority queue + depth penalty）
```

確度バケット:
- **HIGH**: score ≥ 0.75, depth ≤ 3, heuristic edge なし
- **MED**: 0.45 ≤ score < 0.75
- **LOW**: 0.18 ≤ score < 0.45

**Opus評価**: depth penalty `0.93^(depth-2)` は妥当。depth 8で score 0.66倍になるので、遠い候補は自然に落ちる。LOW候補は「念のため確認リスト」として出す — これが「7割のメンタルモデル」の実体。

### 8.3 プロトタイプ実装計画

| 項目 | 仕様 |
|------|------|
| 対象言語 | TypeScript 1言語（大里LMS = Next.js/TS） |
| 構文解析 | Tree-sitter（TypeScript parser完備） |
| ストレージ | SQLite（試作） → PostgreSQL（本番） |
| 実装言語 | Python |
| CLI | index / query / explain / dump の4コマンド |
| 工数 | 28-40時間（4-5週間） |
| やらないこと | GUI、IDE plugin、Neo4j、TypeScript decorators完全解決 |

**Opus評価**: 4-5週間は大里LMS開発と並行して可能。大里LMSがNext.js/TypeScriptなので、**LMS開発とプロトタイプ検証を同時に回せる**（V4のLMS 25ケースがまさにこれ）。さらにPrismaスキーマからresource_usage（テーブル・列レベルの依存）を自動抽出できるため、V5原案のJavaよりも有利。

### 8.4 V5が年収に与える影響

直接的には影響なし。V5は「どう作るか」であって「いくら稼げるか」ではない。

ただし間接的に重要:
- プロトタイプが28-40時間で作れるなら、**Gate 1（15ケース検証）まで最速2ヶ月**
- Gate 1を5月末に通過できれば、Go/No-Goを8月末に前倒しできる
- → 外販開始が2026Q4の早い段階になり、年収ロードマップがやや前倒し

**TypeScript確定によるアドバンテージ**（3/20追記）:
- V5原案のJavaより**Prisma ORMがresource_usage抽出を容易にする**（スキーマファイルに全テーブル・リレーション定義あり）
- 大里LMSにCPDDが既に適用済み → wave_config/アノテーション等の実データでプロトタイプを即検証可能
- Next.jsのファイルベースルーティング → nodeの自動分類（API route vs page vs component）が構文解析不要で判定可能
- LMS開発とCPDD開発を**文字通り同一リポジトリで並行実行中** → 開発コストが相互に吸収される

### 8.5 性能要件（数千万STEPでの設計目標）

| 処理 | 目標時間 |
|------|---------|
| 初回メタデータ生成（10M LOC） | 4-10時間 |
| 初回メタデータ生成（30M LOC） | 12-36時間 |
| 差分更新（1000ファイル） | 5-30分 |
| 影響追跡クエリ（1件） | 5-20秒 |

→ エンタープライズ規模でも実用的。
