# タスク

**CPDD (Change-Propagation-Driven Development)** という新しいAI駆動開発方法論のコアアーキテクチャを、本質的かつ実装可能なレベルまで深く設計してください。

以下に背景・問題定義・初期設計案を示します。これを批判的に検証し、根本から再設計する覚悟で深く考えてください。初期案に引きずられる必要はありません。

## 背景

### CPDDとは
V字モデル上の任意のノード（要件定義・設計書・コード・テスト仕様等）で発生した「変化点(Change)」が、依存グラフに沿って全ノードへ双方向に自動伝播する開発方法論。

### 3本柱
1. **要件定義 → 設計書自動生成**: 要件+非機能要件+ベストプラクティス → AIが設計パターンを数パターン提示 → Human-in-the-Loop で選択
2. **ドキュメント依存管理 + 変化点伝播**: YAMLでV字モデル全ノードの依存関係を明示 → どこかを変更 → AIが影響範囲を自動特定 → 自動更新 or 選択肢提示
3. **最適コンテキスト管理**: 依存グラフから必要な情報だけを動的にロード → Lost in the Middle回避

### この方法論が解こうとしている核心的な問題

エンタープライズパッケージソフト（数千万ステップ）では、ある箇所を変更したとき影響を受ける全箇所を特定するのが極めて困難。理由：

1. **ソースコードの依存**: import/関数呼び出し/継承 → これは静的解析で追える
2. **設定ファイルの依存**: config値で呼び出し先が実行時に分岐する → 静的解析で追えない
3. **マスターデータの依存**: DBのマスターテーブルの値がビジネスロジックを制御する → コードにはSELECT文しかない
4. **暗黙の慣習**: 「ここを変えたらあそこも変える」という経験則 → どこにも書いてない
5. **フレームワークのDI**: Spring Boot等のDIコンテナが設定ファイルとコードの関係を隠蔽する

熟練エンジニア（20年選手）は「メンタルモデル」でこれを解いている。一から調べるのではなく、頭の中の構造的理解をベースに影響箇所をたどる。

**CPDDが解くべき問い: この熟練エンジニアのメンタルモデルをAIでどう再現するか？ 完璧でなくていい。7割の精度で「弱いメンタルモデル」を提供し、若手エンジニアが安全にシステムを改修できるようにする。**

## 初期設計案（批判的に検証してほしい）

### 統合依存グラフ (UDG: Unified Dependency Graph)

5層の依存を一つのグラフに統合する案：

- Layer 1: ファイル依存（imports, includes）— 自動検出
- Layer 2: コード構造依存（関数呼び出し, 継承）— AST/LSPで自動検出
- Layer 3: 設定依存（config値 → コード挙動の分岐）— LLM推論
- Layer 4: データ依存（マスターテーブル → ビジネスロジック）— LLM推論 + 人間注釈
- Layer 5: 慣習依存（暗黙のルール）— 人間注釈 + git履歴学習

ノード: ファイル, 関数, クラス, 設定キー, DBテーブル, 設計書セクション, テストケース等
エッジ: 型付き（calls, imports, reads_config, switched_by, queries, driven_by, convention, co_changes等）+ 信頼度(0-1) + ソース(auto/llm/human/history)

### ブートストラップ戦略

Phase 1（自動検出, 分単位）→ Phase 2（LLM推論, 時間単位）→ Phase 3（人間検証, 継続的）→ Phase 4（git履歴学習, 継続的）

### 伝播エンジン

変化点を起点にBFS/DFS。エッジの信頼度と距離でスコアリング。スコアに応じてauto_update / suggest_review / warn / info のアクションを決定。

### コンテキスト最適化

U字型アテンションカーブを活用。先頭に変更対象、中間にメタ情報（コード全文ではなく要約）、末尾にチェックリスト。必要に応じてオンデマンドで読み込み。

## あなたへの依頼

以下の観点で深く思考し、設計を出力してください。Heavyモードの思考時間を最大限使ってください。

### A. 初期設計の根本的な検証

1. **UDG（統合依存グラフ）というアプローチは正しいか？** グラフ構造が最適なのか、他のデータ構造（例: ハイパーグラフ、Knowledge Graph、Ontology）がより適切ではないか？
2. **5層のLayer分けは正しいか？** もっと本質的な分類軸があるのではないか？ 境界が曖昧になるケースはないか？
3. **「7割の精度」は達成可能か？** 各Layerでの検出精度の見積もりは妥当か？ ボトルネックはどこか？
4. **ブートストラップ戦略は現実的か？** 数千万ステップのシステムでPhase 2のLLM推論をどう実行するか（コスト、時間、精度）？

### B. Layer 3-5 の設計深堀り

ここがフロンティア。既存研究で解かれていない。

5. **設定駆動の依存（Layer 3）をどう検出するか？** 特にDIコンテナ（Spring, Guice等）、リフレクション、プラグインアーキテクチャの場合。コードパターンの分類と検出戦略。
6. **マスターデータ依存（Layer 4）をどう検出するか？** SQLのSELECT文だけでは不十分。「このマスターの値がビジネスロジックのどの分岐を制御するか」をどう推論するか？
7. **暗黙知（Layer 5）の捕捉方法は？** git履歴のco_changes以外に何があるか？ コードレビューのコメント、障害チケット、Slackログ等の非構造化データからの抽出は可能か？
8. **Layer間の相互作用はどう扱うか？** 例: 設定値がマスターデータの参照先を決定し、そのマスターがコードの挙動を制御する（Layer 3→4→2の連鎖）。

### C. 伝播エンジンの精密設計

9. **伝播の停止条件は？** 全ノードに伝播したら意味がない。深さ制限？ 信頼度閾値？ モジュール境界？
10. **偽陽性（過検出）vs 偽陰性（見落とし）のトレードオフ設計は？** セーフティネットとして見落とし（偽陰性）を最小化すべきだが、過検出が多すぎると使われなくなる。最適なバランスはどこか？
11. **循環依存はどう処理するか？** A→B→C→Aの依存がある場合の伝播ループの防止。
12. **変更の「種類」によって伝播パターンは変わるべきか？** インターフェース変更 vs 実装変更 vs 設定変更 vs データ変更で、伝播の深さ・広さが異なるべきでは？

### D. コンテキスト最適化の精密設計

13. **依存グラフからコンテキストを構成する具体的なアルゴリズムは？** どのノードを含め、どの順序で配置し、各ノードの情報量（全文 vs 要約 vs メタ情報のみ）をどう決定するか？
14. **メタ情報（要約）の生成タイミングと更新戦略は？** コード変更のたびに再生成？ キャッシュ？ 階層的要約？
15. **オンデマンド読み込みの判断基準は？** AIが「中間部分のメタ情報では不十分、全文が必要」と判断する基準は何か？

### E. 実装アーキテクチャ

16. **グラフDBの選択**: Neo4j vs 軽量グラフ（NetworkX + JSON永続化）vs カスタムYAML。エンタープライズ規模でのパフォーマンスとポータビリティのトレードオフ。
17. **CI/CDパイプラインへの統合**: 毎コミットでのPhase 1更新、PRレビュー時の影響分析提示の具体的なワークフロー。
18. **Claude Code Plugin / Agent Skills としての実装**: スキル構成、エージェント構成、フック構成の具体的な設計。
19. **段階的ロールアウト**: 数千万ステップのシステムにゼロから導入する際の現実的なステップ。

### F. 根本的な問い

20. **このアプローチの限界はどこにあるか？** CPDDが原理的に解けない問題は何か？ それはどう補うか？
21. **既存の型システム、形式手法、契約プログラミング等との関係は？** 依存グラフは型システムの拡張と見なせるか？
22. **10年後、LLMのコンテキストウィンドウが無限大になったとしても、CPDDは有用か？** コンテキスト管理が不要になっても、変化点伝播の価値は残るか？
